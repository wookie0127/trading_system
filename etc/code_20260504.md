# 🛠️ Telegram Trading Bot Implementation Summary (2026-05-04)

오늘 진행된 텔레그램 기반 자동 매매 시스템의 주요 업데이트 사항과 기술적 배경을 정리합니다.

## 1. 구조적 동시성 (Structured Concurrency) 적용
### 적용 이유 및 내용
- **기존 방식 (`asyncio`)**: 비구조적 태스크 생성(`create_task`)은 태스크가 부모의 생명주기와 무관하게 동작하여, 에러 발생 시 "좀비 프로세스"가 되거나 자원 누수가 발생할 위험이 있었습니다.
- **개선 방식 (`anyio.TaskGroup`)**: 모든 백그라운드 작업(수익률 트래킹, 메시지 리스너, 개별 신호 처리)을 하나의 `TaskGroup`으로 묶어 관리합니다.
- **이점**: 그룹 내의 한 태스크가 실패하면 다른 모든 태스크가 안전하게 종료(Cancel)되고 에러가 전파되어 시스템의 안정성을 보장합니다.

## 2. 선언적 재시도 로직 (`tenacity`)
### 알고리즘 및 시스템적 고려 사항
- **지수 백오프 (Exponential Backoff)**: 네트워크 에러나 세션 만료 시 즉시 재시도하지 않고 대기 시간을 늘려가며(5초 -> 10초 -> 20초...) 재접속을 시도합니다. 이는 서버 부하를 방지하고 성공 확률을 높입니다.
- **관성 유지**: `AsyncRetrying` 루프를 사용하여 리스너의 핵심 로직을 감쌈으로써, 예기치 못한 예외 상황에서도 봇이 죽지 않고 지속적으로 구동됩니다.

## 3. 모의 투자 (Mock Trading) 아키텍처
### 구현 원리
- **관심사 분리**: `DanteTrader` 내부에 `is_mock` 플래그를 도입하여, 주문 실행부만 `KIS API` 호출 대신 `fetch_price` 시뮬레이션으로 교체했습니다.
- **데이터 일관성**: 실제 매매와 동일하게 `active_trades.json`에 기록하고 트래킹 루프를 돌림으로써, 사용자 입장에서는 실전과 동일한 환경에서 로직을 검증할 수 있습니다.

## 4. LLM 기반 시그널 파싱 (Gemini/Codex)
### 알고리즘
- **Context-aware Parsing**: 기존 정규표현식 기반 파서의 한계(문맥 이해 불가)를 극복하기 위해 LLM을 도입했습니다.
- **Prompt Engineering**: 메시지 본문에서 `Action`, `Stop Loss`, `Confidence`, `Rationale`을 추출하여 엄격한 JSON 형식으로 응답받도록 설계했습니다.
- **하이브리드 전략**: LLM 호출 실패(네트워크, 토큰 제한 등) 시 기존 규칙 기반 파서가 백업으로 작동하도록 구성하여 가용성을 확보했습니다.

## 5. 시스템 운영 환경 (Docker)
### 고려 사항
- **환경 격리**: 텔레그램 세션 데이터(`.session`)와 설정 파일(`config.yaml`)을 볼륨 마운트로 관리하여 호스트 환경과 격리하면서도 상태를 보존합니다.
- **자동 복구**: `restart: unless-stopped` 정책을 적용하여 컨테이너 수준의 장애에 대응합니다.

## 6. 학습 데이터 덤프 (Data Analysis)
### 설계 의도
- **역추적성**: `dump_to_csv` 기능을 통해 과거 추천 내역과 당시의 분석 근거, 차트 이미지 경로를 일괄 추출하여 사용자의 투자 복기 및 학습을 지원합니다.
- **인코딩 처리**: 한국어 깨짐 방지를 위해 `utf-8-sig` 인코딩을 적용했습니다.
