# Trading System 코드베이스 정리 (2026-05-05)

이 문서는 이 저장소를 개발자 관점에서 빠르게 파악하기 위한 요약이다. 특히 데이터 파이프라인, 오케스트레이션, 저장 구조, 그리고 이후 AIOps 관점에서 어떤 최적화를 넣어야 하는지에 집중한다.

## 1. 이 시스템을 한 줄로 요약하면

이 저장소는 단일 전략 엔진이라기보다, 여러 시장 데이터 소스와 운영 자동화를 묶은 **데이터 수집/가공/오케스트레이션 중심의 트레이딩 플랫폼 베이스**에 가깝다.

핵심 축은 아래와 같다.

- `KIS OpenAPI`: 한국 주식 실시간성/정확성이 필요한 데이터 수집
- `Yahoo Finance`: 백필과 대체 소스, 미국장/ETF/최근 30일 1분봉 보강
- `Prefect`: 배치성 파이프라인과 외부 알림을 포함한 오케스트레이션
- `Parquet + PostgreSQL/SQLite`: 시계열과 이벤트성 데이터를 성격에 따라 분리 저장
- `Notifier`: Slack/Discord로 운영 피드백 루프 구성

즉, 구조적으로는 “거래 시스템”이라기보다 **시장 데이터 플랫폼 + 운영 자동화 레이어**로 보는 편이 맞다.

## 3. 전체 데이터 파이프라인 구조

### 3.1 파이프라인 계층 분리

이 저장소는 비교적 명확하게 아래 계층으로 나뉜다.

1. `orchestrator/flow`
2. `collector`
3. `validation`
4. `storage`
5. `notification`

예를 들어 Yahoo 1분봉 배치는 아래 순서로 움직인다.

1. `daily_yahoo_intraday_flow`가 시작
2. KOSPI200 구성종목 확보 시도
3. 실패하면 FDR JSON fallback 사용
4. KR 1분봉 수집
5. US ETF/종목 1분봉 수집
6. 성공/실패 메시지 발송

이 구조는 아주 중요하다. 이유는 수집 로직과 운영 제어를 섞지 않았기 때문이다. 나중에 AIOps를 붙일 때도 flow 수준에서 상태를 관찰하고 collector 수준에서 성능을 튜닝할 수 있다.

### 3.2 저장 전략

저장 전략은 용도별로 다르다.

- 시계열 대용량 데이터: Parquet
- 게시판 글 같은 이벤트성 레코드: PostgreSQL 또는 SQLite

이 분리는 적절하다.

- OHLCV/분봉은 스캔성 읽기가 많고 append + dedup이 중요하다.
- 게시판 데이터는 `nid` 기준 idempotent insert가 중요하고, 질의도 row-level이 자연스럽다.

즉, “분석용 컬럼 포맷”과 “트랜잭션성 이벤트 저장소”를 나눠 쓴 셈이다.

## 4. 가장 중요한 파이프라인: 데이터 파이프라인과 오케스트레이션

### 4.1 KOSPI200 구성종목 파이프라인

관련 파일:

- `src/collectors/kr/kospi200_component_collector.py`
- `src/sync/kospi200_symbols_sync.py`

역할은 두 가지다.

- 당일 기준 KRX에서 실제 구성종목 스냅샷을 가져오는 경로
- KRX 실패 시 FinanceDataReader 기반 JSON 레퍼런스를 사용하는 경로

이중화 의도가 명확하다.

- KRX는 공식성은 높지만 세션/접근성이 까다롭다.
- FDR은 비공식 대체지만 자동화 친화적이다.

즉, 정확도와 운영 안정성 사이의 trade-off를 코드로 흡수한 구조다.

구체 예시:

- `fetch_kospi200_components()`는 KRX referer 페이지를 먼저 방문해 세션을 만든 뒤 JSON API를 친다.
- `daily_yahoo_intraday_flow()`는 이 단계가 실패하면 `sync_kospi200_symbols()`와 `get_symbol_list()`로 fallback 한다.

이 구조가 최적화하는 것:

- 데이터 정확도 최대화
- 외부 사이트 특성 때문에 생기는 파이프라인 중단 최소화

### 4.2 KIS 기반 KOSPI200 1분봉 수집

관련 파일:

- `src/collectors/kr/kospi200_intraday_collector.py`
- `src/pipelines/daily_intraday_orchestrator.py`

이 경로는 한국장 마감 후 하루 전체 1분봉을 수집하는 배치다.

핵심 알고리즘은 **역방향 pagination**이다.

- KIS API는 호출당 최대 30개 캔들만 내려준다.
- 그래서 `15:30:00`부터 시작해 과거로 한 페이지씩 이동한다.
- 매 호출 후 가장 오래된 timestamp를 찾아 `1분 전`으로 cursor를 이동한다.

왜 이렇게 구현했는가:

- KIS의 API 제약 때문에 하루 전체 분봉을 한 번에 받을 수 없다.
- 커서를 시간 기준으로 뒤로 이동시키는 방식이 가장 단순하고 API 의미와 잘 맞는다.

구체 예시:

- `_MARKET_CLOSE_HHMMSS = "153000"`
- `_ROWS_PER_CALL = 30`
- `oldest_time - 1 minute`로 다음 페이지 커서를 계산

최적화 포인트:

- 메모리를 과도하게 쓰지 않도록 종목 단위로 수집하고 즉시 저장한다.
- 각 종목 후 `await asyncio.sleep(_API_DELAY)`로 rate limit 리스크를 줄인다.
- `validate_intraday()`로 중복/결측/비정상 가격 제거 후 저장한다.

이 구조가 최적화하는 것:

- API 호출 제한 대응
- 부분 실패 시 전체 데이터를 날리지 않는 안정성
- 종목 수가 많아도 점진 저장 가능

단점도 있다.

- 현재는 종목별 완전 순차 처리라 전체 수집 시간이 길다.
- 장기적으로는 “소규모 bounded concurrency + adaptive rate limiting”으로 옮기는 것이 좋다.

### 4.3 KIS 기반 실시간/준실시간 intraday sync

관련 파일:

- `src/pipelines/intraday_orchestrator.py`
- `src/pipelines/intraday_sync.py`
- `src/runners/intraday_collector_daemon.py`

이 영역은 하루 전체 백필이 아니라 **최근 구간 동기화**에 가깝다.

`intraday_sync.py`의 핵심 포인트:

- 모든 종목을 한 번씩 훑는다.
- 기존 parquet에서 종목별 마지막 timestamp를 읽는다.
- 마지막 수집 시점이 60초 이내면 skip 한다.

즉, 여기서 쓰는 알고리즘은 **마지막 수집 시각 기반 증분 동기화**다.

왜 좋은가:

- 이미 최신인 종목에 대해 불필요한 API 호출을 줄인다.
- 파이프라인 재실행 시에도 자연스럽게 idempotent하게 동작한다.

구체 예시:

- `get_last_sync_stats(dest)`가 parquet의 `timestamp`, `symbol`만 읽어서 마지막 시점을 계산
- `if last_ts and (datetime.now() - last_ts).total_seconds() < 60: continue`

이 구조가 최적화하는 것:

- API quota 절약
- 재시작 복원력
- 중복 저장 비용 감소

`intraday_collector_daemon.py`는 운영형 polling 루프다.

- 시장 시간대에서만 동작
- 10분 간격 poll
- 종목별 랜덤 지연 부여

여기서 랜덤 지연을 둔 이유는 단순한 “예의”가 아니라, 외부 API에 대한 burst 패턴을 완화하기 위함이다.

### 4.4 Yahoo 기반 일일 1분봉 수집

관련 파일:

- `src/pipelines/daily_yahoo_intraday_orchestrator.py`
- `src/collectors/us/yahoo_finance_collector.py`

이 경로는 운영적으로 아주 중요하다. 이 저장소는 KIS만 보지 않고 Yahoo를 **백업 소스이자 미국 데이터 소스**로 사용한다.

핵심 제약:

- Yahoo 1분봉은 최근 약 30일만 가능
- 요청당 최대 약 7일 chunk

그래서 코드가 chunking을 한다.

- `_MAX_DAYS_PER_CHUNK = 7`
- 여러 심볼을 `_BATCH_SIZE = 50` 단위로 나눠 내려받는다

이 알고리즘의 목적:

- yfinance timeout 감소
- 멀티심볼 다운로드의 장점 유지
- 공급자 제약에 맞춘 안정적 백필

또 하나 중요 포인트는 `_extract_symbol_frame()`이다.

- yfinance는 버전/심볼 수에 따라 컬럼 구조가 달라진다.
- flat columns, multi-index columns, single-symbol 예외 형태를 모두 흡수한다.

즉, 이 함수는 단순 파서가 아니라 **공급자 응답 스키마 변동성 완충 장치**다.

### 4.5 백필 파이프라인

관련 파일:

- `src/pipelines/backfill_orchestrator.py`
- `src/runners/collect_daily_stock.py`
- `src/backfill/intraday_backfill_stock.py`

설계 의도는 명확하다.

- 장기 일봉은 KIS에서
- 최근 30일 내 1분봉은 Yahoo에서

왜 분리했는가:

- KIS는 한국 주식 일봉 히스토리 쪽이 안정적이다.
- Yahoo는 최근 분봉 백필이 쉽다.
- 한 소스에 모든 시간축을 강제로 몰지 않고, 데이터 소스의 장점을 나눠 쓴다.

구체 예시:

- `backfill_daily_task()`는 `years * 260`으로 거래일 수를 근사한다.
- `backfill_stock_intraday()`는 일자별 parquet에 대해 해당 symbol이 이미 있으면 skip 한다.

이 구조가 최적화하는 것:

- 동일 작업 재실행 비용 절감
- 빠른 신규 종목 온보딩
- 최근 데이터와 장기 데이터의 소스 분리

### 4.6 대체 데이터 파이프라인: 네이버 종목 토론방

관련 파일:

- `src/collectors/naver_board/orchestrator.py`
- `src/collectors/naver_board/collector.py`
- `src/storage/data_collector.py`

이 파이프라인은 AIOps보다도 “대체 데이터 플랫폼” 관점에서 볼 가치가 있다.

핵심 알고리즘:

- `last_nid` 기반 증분 수집
- `HTTPX -> 실패 시 Playwright` 2단계 수집
- semaphore 기반 bounded concurrency

왜 좋은가:

- 게시판은 페이지를 끝까지 매번 다 긁는 방식이 비효율적이다.
- `nid`는 단조 증가에 가깝기 때문에 watermark 역할을 한다.
- 빠른 경로와 느리지만 강한 경로를 분리해 운영 효율을 확보한다.

구체 예시:

- 각 종목별로 DB에 저장된 최신 `nid`를 읽음
- 페이지를 순회하다 `p["nid"] <= last_nid`를 만나면 중단
- `MAX_CONCURRENCY = 8`로 전체 symbol sweep을 제한

이 구조가 최적화하는 것:

- 네트워크 비용 절감
- 중복 저장 방지
- 차단/렌더링 이슈 대응

이 패턴은 다른 웹 수집기에도 그대로 재사용 가능하다.

### 4.7 뉴스 파이프라인

관련 파일:

- `src/news/daily_news_orchestrator.py`
- `src/news/gemini_news_orchestrator.py`

이건 시장 데이터와는 다르지만, 운영 관점에서는 중요한 “정보 ingestion + 요약” 파이프라인이다.

패턴은 아래와 같다.

- RSS 수집
- 링크 기준 dedup
- 시간순 정렬
- LLM 또는 CLI 요약
- Markdown 아카이브 저장

특히 `dedupe_key = item.link or item.title`는 단순하지만 실전형이다. 뉴스 공급자는 동일 기사 재노출이 잦기 때문에, downstream LLM 비용을 줄이는 데 효과적이다.

## 5. 저장 구조와 알고리즘 의도

### 5.1 Parquet writer

관련 파일:

- `src/storage/parquet_writer.py`

여기는 생각보다 중요한 인프라성 코드다.

핵심 역할:

- 데이터 타입 정규화
- 기존 파일과 병합
- `timestamp + symbol` 기준 dedup
- 날짜별 parquet 저장

왜 이렇게 했는가:

- 시계열 파이프라인은 재실행이 흔하다.
- 따라서 append-only보다 “merge + dedup”이 안전하다.
- 저장 전에 dtype을 안정화하지 않으면 parquet 스키마가 흔들린다.

구체 예시:

- `_normalize_schema()`에서 timestamp, symbol, 가격 컬럼 dtype 정규화
- `drop_duplicates(subset=["timestamp", "symbol"])`

이 구조가 최적화하는 것:

- idempotency
- parquet schema drift 방지
- downstream 분석 안정성

### 5.2 Validation layer

관련 파일:

- `src/storage/data_validation.py`

이 코드는 화려하지 않지만 품질 게이트 역할을 한다.

현재 검증 항목:

- missing timestamp
- duplicate timestamp
- price <= 0
- volume < 0

설계 관점에서 중요한 점은 “예외를 던져 중단”보다 “정제 후 경고 로그”를 택했다는 점이다.

이는 운영 철학상 **완전성보다 가용성 우선**에 가깝다.

즉, 일부 이상치 때문에 하루 전체 수집을 실패시키지 않으려는 선택이다.

### 5.3 이벤트 데이터 저장소

관련 파일:

- `src/storage/data_collector.py`

이 파일은 시계열 수집기보다는 “이벤트/레코드 저장 abstraction”에 가깝다.

특징:

- `TRADING_DATABASE_URL`가 있으면 PostgreSQL
- 없으면 SQLite
- 게시판 데이터는 `INSERT OR IGNORE` 또는 equivalent upsert 성격
- 일봉 데이터는 key 기반 replace

이 구조의 장점:

- 로컬 개발은 SQLite로 빠르게 시작
- 운영은 PostgreSQL로 전환 가능

즉, 개발 생산성과 운영 확장성을 동시에 확보하려는 구조다.

## 6. 오케스트레이션 설계에서 봐야 할 포인트

### 6.1 Prefect를 쓰는 이유

관련 파일:

- `prefect.yaml`

이 저장소에서 Prefect는 단순 cron 대체가 아니다.

Prefect가 제공하는 실질적 가치:

- 배포 정의의 명시성
- 재시도 정책
- 흐름 단위 상태 추적
- 수동 실행/재실행 용이성
- worker / server 분리

현재 등록된 주요 배포:

- `Daily-Yahoo-Intraday`
- `Naver-Board-Sync`
- `Daily-News-Summary`
- `Daily-News-Summary-Gemini`

좋은 점:

- 스케줄과 entrypoint가 코드 외부 `prefect.yaml`에 모여 있다.
- 운영자가 “무슨 잡이 언제 도는지”를 빨리 파악할 수 있다.

아쉬운 점:

- deployment별 `concurrency_limit`, `tags`, `version`이 거의 비어 있다.
- AIOps 관점에서는 run classification 정보가 더 필요하다.

### 6.2 알림 설계

관련 파일:

- `src/bots/notifier.py`

이 시스템은 거의 모든 주요 flow에서 시작/성공/실패 메시지를 보낸다.

이는 단순 편의 기능이 아니라 운영 관측성의 가장 기본 레이어다.

다만 현재는 한계도 있다.

- 구조화된 메트릭보다 텍스트 메시지 중심
- error taxonomy가 약함
- run_id, duration, row_count 같은 표준 필드가 통일되어 있지 않음

앞으로는 알림 메시지와 메트릭을 분리하는 편이 좋다.

## 7. 인프라 관점에서 좋은 구조

관련 파일:

- `docker-compose.yml`

현재 compose는 역할 분리가 잘 되어 있다.

- `prefect-server`
- `prefect-worker`
- `postgres`
- `postgres-backup`
- `api`
- `collector`
- `tleading-bot`

이 구조의 장점:

- 오케스트레이션 서버와 실행 worker 분리
- Postgres 백업 컨테이너 분리
- 데이터 수집기와 API 분리

이건 앞으로 AIOps를 얹기 좋은 구조다. 이유는 관찰 대상이 서비스 단위로 이미 나뉘어 있기 때문이다.

## 8. 지금 코드에서 보이는 설계 철학

이 저장소 전반에는 몇 가지 반복되는 설계 철학이 보인다.

### 8.1 완벽한 단일 소스보다 다중 소스 복원력

예시:

- KRX 실패 시 FDR fallback
- KIS 외에 Yahoo 사용
- Naver HTTPX 실패 시 Playwright fallback

이건 트레이딩 시스템에서는 좋은 방향이다. 외부 데이터 공급자는 항상 깨질 수 있기 때문이다.

### 8.2 재실행 가능한 파이프라인

예시:

- parquet merge + dedup
- `last_nid` watermark
- symbol 존재 여부 기반 backfill skip
- 마지막 timestamp 기반 intraday sync skip

즉, 대부분의 잡이 idempotent 또는 준-idempotent하게 설계되어 있다.

### 8.3 실전형 운영 우선

예시:

- Slack/Discord 즉시 알림
- Docker Compose 운영
- 백업 컨테이너
- Prefect 기반 스케줄링

이 프로젝트는 연구용 스크립트 묶음이 아니라, 이미 운영성을 상당히 의식한 구조다.

## 9. 앞으로 AIOps 구축을 위한 최적화 방법론

여기서부터가 가장 중요하다. 지금 구조는 괜찮지만, AIOps를 붙이려면 “관측 가능한 파이프라인”으로 더 명확하게 바꿔야 한다.

### 9.1 방법론 1: 모든 파이프라인에 표준 실행 메타데이터를 붙이기

각 flow/task에서 최소한 아래 필드를 공통으로 남기는 것이 좋다.

- `pipeline_name`
- `run_id`
- `source`
- `target_date`
- `started_at`
- `finished_at`
- `duration_ms`
- `records_fetched`
- `records_written`
- `records_deduped`
- `error_type`
- `status`

구현 방법:

- `src/common` 또는 `src/observability`에 `RunContext`, `PipelineMetrics` 같은 dataclass 생성
- 각 collector가 결과를 `dict`가 아니라 typed result로 반환
- Prefect task/flow와 notifier에서 같은 메타데이터를 재사용

예시 방향:

- `collect_kospi200_intraday()`가 단순 `None` 대신
  `{"symbols_total": 200, "symbols_success": 196, "rows_written": 73124, "rows_invalid": 12}` 반환

이렇게 해두면 AIOps에서 이상탐지 기준을 만들기 쉽다.

### 9.2 방법론 2: watermark와 lineage를 1급 개념으로 승격

현재도 watermark 개념은 있다.

- `last_nid`
- 마지막 timestamp
- target_date parquet 존재 여부

하지만 산발적으로 흩어져 있다.

개선 방향:

- `pipeline_state` 테이블을 두고 각 파이프라인의 마지막 성공 시점, 마지막 성공 watermark, 마지막 row count를 저장

예시 스키마:

- `pipeline_name`
- `partition_key`
- `watermark_value`
- `last_success_at`
- `last_status`
- `stats_json`

구현 이점:

- 재실행 기준을 파일 존재 여부에만 의존하지 않음
- AIOps 룰 작성이 쉬워짐
- “어제는 200 종목 성공했는데 오늘은 123 종목만 성공” 같은 drift 감지 가능

### 9.3 방법론 3: 수집 결과를 데이터 품질 규칙으로 정형화

현재 validation은 함수형으로 존재한다. 다음 단계는 이를 “룰 엔진화”하는 것이다.

예:

- KR 1분봉: 종목당 최소 분봉 수가 300개 미만이면 warning
- 마감 후 수집: 장 마감일 기준 KOSPI200 종목 수가 200이 아니면 warning
- 뉴스: dedup 후 기사 수가 5개 미만이면 degraded
- 게시판: 수집 종목 수 대비 신규 글 0건이 비정상적으로 길게 지속되면 warning

구현 방법:

- `src/storage/data_validation.py`를 rule registry 구조로 확장
- rule 실행 결과를 Prefect artifact 또는 DB에 저장

### 9.4 방법론 4: concurrency를 “고정값”이 아니라 “적응형 제어”로 바꾸기

현재는 다음과 같은 고정값이 많다.

- `_API_DELAY = 0.2`
- `MAX_CONCURRENCY = 8`
- `_BATCH_SIZE = 50`

초기에는 맞는 방식이지만, 운영이 커지면 adaptive control이 필요하다.

예시:

- 최근 10분 에러율이 높으면 concurrency 자동 감소
- 외부 API latency가 증가하면 sleep 증가
- 특정 source만 불안정하면 fallback source로 자동 전환

구현 방법:

- 최근 실행 결과를 `pipeline_state` 또는 metrics store에 저장
- 다음 실행 시 이전 상태를 읽어 동적으로 파라미터 결정

이게 AIOps의 시작점이다. “모니터링”이 아니라 “실행 파라미터 자동 조절”로 넘어가야 한다.

### 9.5 방법론 5: 텍스트 알림에서 구조화된 관측성으로 이동

지금은 Slack/Discord 메시지가 중심이다. 앞으로는 아래를 병행해야 한다.

- 구조화 로그 JSON
- 시계열 메트릭
- 파이프라인 실행 이벤트 저장

추천 구현 순서:

1. Loguru 로그를 JSON 포맷으로도 남기기
2. Prefect run metadata를 Postgres에 적재
3. Prometheus exporter 또는 OpenTelemetry 도입
4. Grafana 대시보드 구성

최소 대시보드 예시:

- 파이프라인별 성공률
- 파이프라인별 실행 시간
- 일자별 row count
- 소스별 에러율
- fallback 사용 빈도
- 종목별 데이터 누락률

### 9.6 방법론 6: 데이터 계약(Data Contract) 만들기

현재 코드에는 암묵적 스키마가 많다. 예를 들어 parquet row는 사실상 계약이지만 문서화된 계약은 약하다.

다음 계약을 명시하는 것이 좋다.

- `kr/kospi200/1min/<date>.parquet`의 필수 컬럼
- timestamp timezone 규칙
- symbol 표현 규칙
- null 허용 범위
- dedup key

구현 방법:

- `schemas/` 또는 `src/contracts/` 디렉토리 추가
- `pandera`, `pydantic`, `pyarrow schema` 중 하나로 계약 정의

이 작업은 AIOps에서 중요하다. 이상징후 탐지는 “정상 스키마” 정의가 있어야 가능하기 때문이다.

## 10. 이 저장소에서 빠르게 개선 효과가 큰 구현 우선순위

### 1순위: 파이프라인 실행 결과 표준화

바로 효과가 난다.

- flow/task 반환값을 구조화
- 성공/실패/건수/지연시간 통일
- notifier 메시지도 이 값을 사용

### 2순위: Postgres에 pipeline_run / pipeline_state 테이블 추가

현재 이미 Postgres가 있으므로 비용 대비 효과가 크다.

추천 테이블:

- `pipeline_runs`
- `pipeline_states`
- `data_quality_results`

### 3순위: parquet write 통계 수집

현재 `write_parquet()`는 몇 행 저장했는지는 로그로 남기지만, merge 전후 row 변화나 dedup 건수는 노출하지 않는다.

개선 예시:

- `input_rows`
- `existing_rows`
- `merged_rows`
- `dedup_removed_rows`

이 값은 AIOps 지표로 매우 유용하다.

### 4순위: fallback 발생을 1급 이벤트로 승격

예:

- KRX 실패 후 FDR fallback
- HTTPX 실패 후 Playwright fallback

지금은 로그/메시지에만 남지만, 앞으로는 이벤트 테이블에 남겨야 한다.

이유:

- fallback 빈도 상승은 외부 공급자 품질 저하 신호다.
- 장기적으로 source reliability score를 만들 수 있다.

### 5순위: 파이프라인별 SLO 정의

예시:

- Daily Yahoo Intraday: 오전 07:00 전 완료율 99%
- KOSPI200 1분봉 일배치: 장 마감 후 30분 내 95% 종목 수집
- Naver Board: 종목별 12시간 주기 내 1회 이상 성공

SLO가 있어야 AIOps 자동화의 목표 함수가 생긴다.

## 11. 개발자가 특히 유심히 볼 만한 “좋은 구현” 예시

### 예시 1. parquet 기반 idempotent 저장

`src/storage/parquet_writer.py`는 단순 저장기가 아니다.

- schema normalize
- 기존 파일 read
- concat
- dedup
- rewrite

재실행이 흔한 데이터 파이프라인에서는 매우 실용적이다.

### 예시 2. 외부 API 제약을 코드 구조로 흡수

`src/collectors/kr/kospi200_intraday_collector.py`의 역방향 pagination,
`src/collectors/us/yahoo_finance_collector.py`의 7일 chunking은
“API가 이래서 어쩔 수 없이”가 아니라, 제약을 흡수한 설계다.

### 예시 3. fallback을 전략적으로 사용

`daily_yahoo_intraday_flow()`와 `NaverBoardCollector.collect_symbol()`은
“실패하면 끝” 대신 “실패하면 다음 경로 시도” 구조다.

운영형 시스템에서 굉장히 중요한 패턴이다.

### 예시 4. 개발/운영 저장소 분리 전략

`DataCollector`가 SQLite와 PostgreSQL을 모두 지원하는 방식은
초기 개발 속도와 운영 이행성을 동시에 잡는다.

## 12. 지금 코드에서 기술 부채로 볼 수 있는 부분

빠르게 읽고 주의할 만한 지점도 있다.

### import path 처리 중복

여러 파일이 `sys.path.insert()` 또는 `sys.path.append()`를 반복한다.

문제:

- 실행 방식에 따라 import 동작이 달라질 수 있음
- 테스트/패키징이 불안정해짐

개선:

- `src` 패키지 기준 실행을 고정
- CLI entrypoint 또는 proper package install 방식으로 통일

### flow/task 반환값의 표준 부재

일부는 `True`, 일부는 `tuple`, 일부는 `dict`, 일부는 `None`이다.

문제:

- 관측성 자동화가 어려움
- 재사용성이 떨어짐

개선:

- typed result 모델로 통일

### 순차 처리 비중이 높음

특히 KIS 계열 수집은 대부분 순차적이다.

문제:

- 대상 종목 수가 늘면 총 소요시간이 빠르게 증가

개선:

- semaphore 기반 제한적 동시성
- adaptive backoff

### 구조화 메트릭 부족

문제:

- 실패는 보이지만 “왜 점점 느려지는지”, “언제부터 데이터량이 줄었는지”는 자동 판단하기 어려움

개선:

- metrics first 설계 추가

## 14. 결론

이 저장소의 강점은 아래 세 가지다.

- 외부 데이터 공급자 제약을 현실적으로 흡수하는 구조
- 재실행 가능한 파이프라인과 운영 알림이 이미 들어간 구조
- AIOps로 확장할 수 있는 서비스 분리와 오케스트레이션 기반이 이미 있음

앞으로의 핵심 과제는 “동작하는 수집기”를 “관측 가능하고 스스로 조정 가능한 파이프라인”으로 바꾸는 것이다.

가장 먼저 해야 할 일은 화려한 AI 도입이 아니라 아래다.

1. 파이프라인 결과 표준화
2. watermark/state 저장소 도입
3. 구조화 메트릭 수집
4. data quality rule 정형화
5. adaptive concurrency / fallback 정책화

이 다섯 가지가 정리되면, 그 다음부터는 anomaly detection, auto-remediation, source health scoring 같은 AIOps 기능을 훨씬 빠르게 얹을 수 있다.
